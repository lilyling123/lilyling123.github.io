    线程安全性

    编写线程安全的代码，核心在于对状态访问操作进行管理，特别是对共享的和可变的状态的访问

什么是线程安全性

某个类的行为与其规范完全一致

无状态对象一定是线程安全的

原子性

count++这样的自增操作不是线程安全的

竞态条件

当某个计算的正确性取决于多个线程的交替执行程序时，那么就会发生竞态条件
最常见的竞态条件类型就是“先检查后执行”操作

示例：延迟初始化中的竞态条件

加锁机制

内置锁

synchronized(object){
}

重入

如果某个线程试图获得一个已经是由它自己持有的锁，那么这个请求就会成功
重入意味着获取锁的操作的粒度是“线程”而不是“调用”

用锁来保护状态

需要自行构造锁协议或者同步策略来实现对共享状态的安全访问，并且在程序中自始至终的使用它们

每个共享的和可变的变量都应该只由一个锁来保护，从而使维护人员知道是哪一个锁

活跃性与性能

使用同步代码块缩小锁范围可显著的改善性能

当执行较长时间较长的计算或者可能无法快速完成的操作时，一定不要持有锁

对象的共享

可见性

为了确保多个线程之间对内存的写入操作的可见性，必须使用同步机制

只要有数据在多个线程之间共享， 就使用正确的同步

失效数据

读线程可能看到的是一个失效数据，除非每次访问变量时都使用同步

非原子的64位操作

当线程在没有同步的情况下读取变量时，可能会得到一个失效值，但至少这个值是由之前某个线程设置的值，而不是一个随机值，也被称为最低安全性。但是不适用于double和long，JVM允许将64位的读操作或写操作分解为两个32位的操作，那么可能会读到某个值的高32位和另一个值的低32位

加锁与可见性

如果同步，则可见

Volatile变量

轻量级的同步操作，也是可见的

发布与逸出

发布：使对象能够在当前作用域之外的代码中使用
当某个不应该发布的对象被发布时，这种情况就被称为逸出

当发布一个对象时，在该对象的非私有域中引用的所有对象同样会被发布——如果一个已经发布的对象能够通过非私有的变量引用和方法调用到达其他的对象，那么这些对象也会被发布。

安全的对象构造过程

不要再构造过程中使this引用逸出

线程封闭

如果仅在单线程内访问数据，就不需要同步，这种技术被称为线程封闭
Swing和JDBC

Ad-hoc线程封闭

维护线程的封闭性的职责完全由程序实现来承担，比较脆弱，少用

栈封闭

栈封闭是线程封闭的一种特例：就是将变量定义在方法中

ThreadLocal类

维护线程封闭性的一种更规范的方法是使用ThreadLocal

不变性

不可变对象一定是线程安全的

Final域

final域能确保初始化过程的安全性，可以不受限制的访问不可变对象

使用Volatile类型来发布不可变对象

安全发布

不正确的发布：正确的对象被破坏

使用静态化初始器
